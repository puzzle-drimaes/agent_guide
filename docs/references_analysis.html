1ECC vs k-sdd — 한눈에

ECC 넓이

  • ~10개 툴(어댑터 11) 대상 배포 엔진 + 방대한 운영 자산
  • 강점: 어댑터·3계층 매니페스트·provenance·공급망 보안
  • 역할: 운영 자산 & 하네스 패턴 라이브러리

k-sdd 깊이

  • 경량 설치 엔진 + Spec-Driven 방법론
  • 강점: manifest→plan→apply·path-safety·자율 구현 루프
  • 역할: 설치 엔진 뼈대 & SDD 워크플로우
축ECCk-sdd
배포 인프라 넓이강함(어댑터 11)경량(8 타깃)
설치 엔진 명료성대규모·복잡선명·이식 쉬움
자율 구현 방법론오케스트레이션 가이드impl/review/debug 루프
공급망 보안스캔·유니코드·서명 관점path-safety 중심
승인 게이트GATE 마커spec.json 상태머신
결론: k-sdd = 설치 엔진 뼈대, ECC = 자산·패턴·보안 라이브러리. 사내 시스템은 둘을 합쳐 agent-deploy로 구현한다.

2통합 배포 아키텍처 — "하나의 정의 → N개 툴"

정규 소스를 설치 타임에 각 툴 네이티브 설정으로 변환하는 4단 파이프라인(두 프로젝트 공통).

5agent-deploy 현황 (이미 구현 vs 남은 것)

본 repo agent-deploy/에 구현된 것과 미구현. ✅는 스모크 테스트 수준에서 기본 동작만 확인(프로덕션 검증 아님), 자산 항목은 파일 설치·스코핑까지를 의미.

✅ 구현됨

  • 매니페스트(modules/profiles) + 의존성 해석·순환 검출
  • 타깃 어댑터(claude/cursor/codex/gemini) + 서브경로·단일파일 스코핑
  • install-state provenance + 스키마
  • 비파괴 deepMerge + path-safety + symlink 가드
  • 계획/적용 분리 + dry-run + --json
  • scope: project 기본 + --global(home)
  • skip+reason 연산, op에 scope/reason 기록
  • 아키텍처/SDD/commit convention/core common 룰 + developer workflow skill 자산 설치
  • product/business/governance profile과 직무별 skill·prompt 스타터 자산
  • Codex TOML add-only merge + AGENTS/GEMINI managed block append-markdown
  • 번들 스크립트 install.sh/install.bat + CI 불변식 검증(validate 4종 가드, smoke test 19개)

6주의할 점

  • 범위 과다 — ECC 전체(271 skills)는 20명 규모엔 무겁다. starter set부터.
  • 하네스 기능 차이 — hook/skill/MCP 지원이 도구마다 다름 → semantic equivalence + skip+reason 기록.
  • SDD 과잉 — requirements/design/tasks를 전 업무 강제 시 adoption↓. sdd는 선택 프로파일.
  • spec = contract — 진실의 원천은 코드+테스트. spec은 부분 간 계약일 뿐.
  • MCP 보안 — filesystem/npx MCP는 기본 제외, allowlist + 버전 고정.
  • Hook 강제 수준 — warning/soft → 전사 rollout 후 required로 점진.
  • install-state는 기반 — 단순 기록이 아니라 update/repair/uninstall의 토대로 설계.

7통합 결론

사내 표준 배포 시스템 = agent-deploy ├─ 설치 엔진 뼈대 ← k-sdd (manifest→plan→apply, path-safety, bundle) ├─ 자산·패턴·보안 라이브러리 ← ECC (어댑터, provenance, 공급망 스캔, 룰/스킬/에이전트) ├─ 자율 구현 방법론 ← k-sdd (게이트, 경계우선, 핸드오프, 적대적 리뷰) └─ 거버넌스(6룰/KPI) ← 사내 (PR노하우·회고·프롬프트DB↔스킬 승격)
한 줄: ECC의 넓은 배포 인프라 + k-sdd의 선명한 설치 엔진·자율 구현 루프를, project 기본 / 전역 옵션의 번들로 사내 2~3개 툴에 축소 이식한다. 골격(agent-deploy)과 1차 Codex/Gemini 어댑터·직무별 자산은 이미 동작하며, 남은 것은 capability matrix·번들 배포·보안·생애주기와 Pilot 운영 준비다.
