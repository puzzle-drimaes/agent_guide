# Prompt DB 샘플 엔트리 (seed 5종)

[15-prompt-db-operations.md](plans/codex/company-wide-agent-rollout/15-prompt-db-operations.md)의 필드 정의에 맞춘 예시 데이터다.
Notion Prompt DB를 만들 때 이 5개를 그대로 시드(seed)로 넣을 수 있다.

```text
- 운영 필드(성공률 / 사용 횟수 / 사용자 수)는 사용되면서 채워지므로 "(운영 중 집계)"로 둔다.
- 작성자 / Owner는 실제 담당자로 바꾼다(아래는 seed 기본값).
- 프롬프트 본문은 회사 skeleton(역할 / 목적 / 대상 / 배경 / 제약조건 / 출력 형식)을 따른다.
- 출처/라이선스는 사내 원본이므로 internal로 둔다.
```

## 1. 회의 요약

```text
이름: 회의 요약 정리
목적: 회의록/녹취를 결정·액션아이템·리스크·미결로 구조화한다.
대상 직무: business, product
AI 도구: Claude
권장 profile: business
상태: active
작성자: ai-governance(seed)
Owner: 사업/운영 리드
Reviewer: ai-governance
성공률: (운영 중 집계)
사용 횟수: (운영 중 집계)
사용자 수: (운영 중 집계)
출처/라이선스: 사내 원본 / internal
```

프롬프트 본문:

```text
## 역할
당신은 회의 진행과 기록에 능숙한 운영 매니저이다.

## 목적
아래 회의 내용을 의사결정 / 액션아이템 / 리스크 / 미결 질문으로 정리한다.

## 대상
결과를 읽을 사람: 회의 비참석자를 포함한 팀 전원

## 배경
[회의록 또는 녹취. 고객명·사번 등 민감정보는 <고객명>, <사번>으로 치환한다.]

## 제약조건
- 회의에서 실제로 나온 내용만 쓴다(추측 금지, 필요하면 "확인 필요"로 표시).
- 액션아이템은 담당자와 기한을 함께 적는다(없으면 "미정").
- 외부 자료를 인용하면 출처를 표기한다.

## 출력 형식
1. 한 줄 요약
2. 결정 사항
3. 액션아이템 (담당자 / 기한)
4. 리스크
5. 미결 질문
```

입력 예시:

```text
"오늘 회의: <고객명> 대응 일정 논의. 김 PM이 다음 주 금요일까지 1차 답변안 작성하기로.
가격은 아직 미확정. 법무 검토 필요하다는 의견 있었음."
```

출력 예시:

```text
1. 한 줄 요약: <고객명> 대응 일정과 답변안 작성 담당을 정함, 가격/법무는 미결.
2. 결정 사항: 1차 답변안 작성 진행.
3. 액션아이템: 1차 답변안 작성 — 김 PM / 다음 주 금.
4. 리스크: 가격 미확정으로 답변 범위 제한 가능.
5. 미결 질문: 가격 확정 시점? 법무 검토 필요 범위?
```

## 2. 고객 응답 초안

```text
이름: 고객 응답 초안
목적: 고객 문의에 정중하고 정확한 답변 초안을 만든다(사람 검토 전 단계).
대상 직무: business
AI 도구: Claude, Gemini
권장 profile: business
상태: active
작성자: ai-governance(seed)
Owner: CS/운영 리드
Reviewer: ai-governance
성공률: (운영 중 집계)
사용 횟수: (운영 중 집계)
사용자 수: (운영 중 집계)
출처/라이선스: 사내 원본 / internal
```

프롬프트 본문:

```text
## 역할
당신은 침착하고 정확한 고객 응대 담당자이다.

## 목적
아래 고객 문의에 대한 답변 초안을 작성한다. 최종 발송 전 사람이 검토한다.

## 대상
결과를 읽을 사람: 문의한 고객

## 배경
[고객 문의 원문. 고객 개인정보는 <고객명>, <주문번호> 등으로 치환한다.]
[확정된 사실 / 미확정 사항을 함께 적는다.]

## 제약조건
- 확정되지 않은 내용은 단정하지 않는다("확인 후 안내드리겠습니다").
- 사과/안내/다음 단계를 분명히 한다.
- 과장·약속 남발 금지. 민감정보는 답변에 그대로 노출하지 않는다.
- 톤: 정중하고 간결하게.

## 출력 형식
1. 답변 초안(그대로 보낼 수 있는 형태)
2. 검토 포인트(사람이 확인해야 할 부분)
```

입력 예시:

```text
"고객 <고객명>이 주문 <주문번호> 배송이 지연된다고 항의. 실제 지연 맞고, 보상 정책은 미확정."
```

출력 예시:

```text
1. 답변 초안:
   "<고객명>님, 주문 <주문번호> 배송이 지연되어 불편을 드려 죄송합니다.
    현재 배송 상태를 확인하고 있으며, 보상 가능 여부는 확인 후 빠르게 안내드리겠습니다."
2. 검토 포인트: 실제 배송 예정일, 보상 정책 확정 여부.
```

## 3. PRD 초안

```text
이름: PRD 초안 작성
목적: 아이디어/요청을 PRD(문제·목표·요구사항·비목표·완료조건) 초안으로 정리한다.
대상 직무: product
AI 도구: Claude, Codex
권장 profile: product
상태: active
작성자: ai-governance(seed)
Owner: 기획/PM 리드
Reviewer: ai-governance
성공률: (운영 중 집계)
사용 횟수: (운영 중 집계)
사용자 수: (운영 중 집계)
출처/라이선스: 사내 원본 / internal
```

프롬프트 본문:

```text
## 역할
당신은 제품 기획(PM)이다.

## 목적
아래 아이디어/요청을 PRD 초안으로 정리한다.

## 대상
결과를 읽을 사람: 개발/디자인/사업 등 협업 팀

## 배경
[아이디어 또는 고객/내부 요청. 미공개 로드맵·계약 정보는 <비공개>로 치환한다.]

## 제약조건
- 추측이 필요하면 가정을 먼저 명시한다.
- 측정 가능한 완료 조건(acceptance criteria)을 쓴다.
- 범위에 넣지 않을 것(비목표)을 분명히 한다.

## 출력 형식
1. 문제 정의
2. 목표 / 성공 지표
3. 핵심 요구사항(우선순위)
4. 비목표(Non-goals)
5. 완료 조건(acceptance criteria)
6. 열린 질문
```

입력 예시:

```text
"사용자가 공유 폴더에서 .md를 골라 자기 프로젝트에 쉽게 적용하게 하고 싶다."
```

출력 예시:

```text
1. 문제 정의: 공유된 .md를 적용하는 과정이 수동이라 진입장벽이 있다.
2. 목표: 적용까지 3단계 이내, 비개발자도 가능. 지표: 적용 성공 사례 수.
3. 핵심 요구사항: (P0) externals 자동 검토/배치, (P1) 충돌 안내, (P2) 적용 이력.
4. 비목표: 신규 .md 자동 생성, 원격 동기화.
5. 완료 조건: 비개발자 1명이 도움 없이 .md 1개를 적용한다.
6. 열린 질문: 충돌 시 기본 동작은?
```

## 4. 개발 구현 요청

```text
이름: 개발 구현 요청
목적: 구현 요구사항을 설계·코드·설명·개선점으로 정리한다.
대상 직무: developer
AI 도구: Codex
권장 profile: developer
상태: active
작성자: ai-governance(seed)
Owner: 개발 리드
Reviewer: ai-governance
성공률: (운영 중 집계)
사용 횟수: (운영 중 집계)
사용자 수: (운영 중 집계)
출처/라이선스: 사내 원본 / internal
```

프롬프트 본문:

```text
## 역할
당신은 Senior Software Engineer이다.

## 목적
아래 요구사항을 구현한다.

## 대상
결과를 읽을 사람: 리뷰어 및 유지보수 개발자

## 배경
[요구사항. credential/token/실데이터는 절대 넣지 않고 <ENV_VAR> 등으로 표기한다.]
- 개발환경: OS / 언어 / 프레임워크 / 버전

## 제약조건
- 유지보수성·성능·예외처리를 고려한다.
- 핵심 로직에 주석을 단다.
- 회사 아키텍처/커밋 규칙을 따른다(해당 프로필 설치 시).
- 가정이 필요하면 먼저 밝힌다.

## 출력 형식
1. 설계(접근/핵심 결정)
2. 코드
3. 설명
4. 개선점/후속 과제
```

입력 예시:

```text
"공유 폴더에서 받은 .md의 frontmatter 유효성을 검사하는 함수.
언어: Node.js. 누락 필드를 에러로 모아 반환."
```

출력 예시:

```text
1. 설계: 필수 키 목록과 입력 비교, 누락/타입 오류를 배열로 수집.
2. 코드: function validateFrontmatter(fm) { ... }
3. 설명: 닫힌 스키마로 unknown 키도 오류 처리.
4. 개선점: enum 값 검증, asset_type별 조건부 필수 추가.
```

## 5. 리서치 요약

```text
이름: 리서치/문서 요약
목적: 긴 자료/검색 결과를 핵심·근거·한계·다음 단계로 요약한다.
대상 직무: developer, product, business, governance
AI 도구: Claude
권장 profile: minimal
상태: active
작성자: ai-governance(seed)
Owner: 직무 리드(공용)
Reviewer: ai-governance
성공률: (운영 중 집계)
사용 횟수: (운영 중 집계)
사용자 수: (운영 중 집계)
출처/라이선스: 사내 원본 / internal
```

프롬프트 본문:

```text
## 역할
당신은 정확성을 중시하는 리서치 분석가이다.

## 목적
아래 자료를 의사결정에 쓸 수 있게 요약한다.

## 대상
결과를 읽을 사람: 시간이 없는 의사결정자

## 배경
[자료 본문 또는 링크/검색 결과. 외부 자료는 출처를 함께 제공한다.]

## 제약조건
- 자료에 없는 내용은 지어내지 않는다(불확실하면 "불충분"으로 표시).
- 주장마다 근거(출처)를 연결한다.
- 외부 자료 사용 시 출처/라이선스를 표기한다(source-attribution 규칙).

## 출력 형식
1. 3줄 요약
2. 핵심 포인트(근거/출처 포함)
3. 한계·불확실한 점
4. 추천 다음 단계
```

입력 예시:

```text
"아래 3개 문서를 비교해 우리 상황에 맞는 방식을 추천해줘. [문서 링크 3개]"
```

출력 예시:

```text
1. 3줄 요약: A는 단순하지만 확장성 약함, B는 반대, C는 절충안.
2. 핵심 포인트: A=빠른 도입(출처1), B=대규모 적합(출처2), C=단계적 전환(출처3).
3. 한계: 비용 데이터 부재로 TCO 비교 불가.
4. 추천 다음 단계: C로 PoC 후 비용 측정.
```
