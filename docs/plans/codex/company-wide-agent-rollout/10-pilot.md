# 10. Pilot 운영

## 목적

전사 배포 전에 작은 그룹에서 설치, profile, 공용 계정 운영, 지식 공유 룰의 문제를 발견한다.
Pilot의 목표는 "완벽한 배포"가 아니라 "전사 rollout 가능/불가를 근거 있게 판단"하는 것이다.

## Pilot 대상

권장 구성 (총 5명):

| 역할 | 인원 | 담당자 | 사용 profile | OS/target |
|---|---|---|---|---|
| 개발자 | 2명 | (지정) | developer | (지정) |
| PM/기획 | 1명 | (지정) | product | (지정) |
| 비개발 직군 | 1명 | (지정) | business | (지정) |
| AI governance 담당 | 1명 | (지정) | governance | (지정) |

선정 기준:

```text
- 비개발 직군은 터미널 경험이 적은 사람을 우선 포함한다(비개발자 UX 검증 목적).
- OS는 Windows/macOS/Linux가 최소 2종 이상 섞이도록 배정한다.
- target은 Codex/Claude를 기본으로, 가능하면 Gemini도 1명에게 배정한다.
- 각 참가자는 Pilot 기간 동안 실제 업무 1건 이상을 agent로 수행할 수 있어야 한다.
```

## 기간 / 일정

```text
총 기간: 2주
```

| 시점 | 일자 | 체크포인트 |
|---|---|---|
| 시작(D0) | (지정) | 킥오프, 설치 가이드 배포, 계정/공유 폴더 접근 확인 |
| 1주차 종료(D+7) | (지정) | 설치·첫 업무 적용 점검, 1차 blocker 회수 |
| 2주차 종료(D+14) | (지정) | 운영·회고·KPI 집계, rollout 판단 회의 |

```text
- 일자는 Pilot 킥오프 시 확정해 위 표에 기입한다.
- 1주차는 설치/첫 업무 적용, 2주차는 운영/지식 공유 검증에 집중한다.
```

## 운영 방식

Pilot은 별도의 리허설 단계를 두지 않는다. 준비가 완료된 bundle/profile을 먼저 Pilot 대상자에게 배포하고, 실제 설치·사용 과정에서 나온 피드백을 즉시 수집해 문서, profile, rule, prompt, skill에 반영한다.

```text
- Pilot 전에는 별도 rehearsal 대신 필수 산출물과 안전장치 준비 여부만 확인한다.
- 배포 후 피드백은 agent가 `.md`로 정리해 Google Drive `AI-Knowhow/feedback/` 업로드를 먼저 유도한다.
- Drive 커넥터가 없으면 agent가 피드백 `.md` 내용을 만들어 주고, 사용자가 Drive 웹에서 업로드한다.
- blocker는 우선 workaround를 문서화하고, 재발 가능성이 높은 항목부터 bundle/profile에 반영한다.
- Pilot 종료 시점에는 반영 완료 항목과 전사 rollout 전 보류 항목을 구분한다.
```

## Pilot 전 준비

```text
- minimal/developer/product/business profile 준비
- Codex/Claude target 동작 확인
- SETUP_WIZARD.md 기반 설치 가이드 초안 작성 및 필수 경로 확인
- Google Drive AI-Knowhow/feedback 폴더와 업로드 권한 준비
- Drive 운영 메모, 공유 폴더, governance 기록 위치 준비
- 공유 폴더(Google Drive 지정 폴더) 준비 + .md 업로드/적용 안내 (1차 도입 메커니즘)
- Prompt DB 템플릿 준비(반복 사용분 정리용, 강제 등록 아님)
- 공용 AI 계정 9개 매핑표 준비(담당자/금지 데이터 포함)
- 회고 질문 준비(아래 "회고 질문"; 제출은 권장, 강제 아님)
```

## 1주차 검증

설치 검증:

```text
- Windows/macOS/Linux 중 가능한 OS에서 설치
- Codex/Claude/Gemini 중 가능한 target에서 설치
- dry-run과 apply 비교
- 설치 실패 시 SETUP_WIZARD.md "설치가 안 될 때" 절차로 자가 복구 가능 여부 확인
```

업무 적용:

```text
- 개발자: 실제 업무 1건을 agent로 수행 (PR 노하우 기록은 권장)
- PM: 요구사항 초안 1개 작성
- 비개발자: 회의록/고객응대/제안서 중 1개 작성
- 각자 쓸 만한 스킬/프롬프트 .md를 공유 폴더에 1개 이상 올려보기 (권장)
```

## 2주차 검증

운영 검증:

```text
- 공유 폴더에 모인 .md를 선택해 자기 프로젝트에 적용해보기 (핵심 신호)
- 주간 회고 (권장, 강제 아님)
- 반복 사용 프롬프트만 Prompt DB 등록 (선택)
- 실패 사례 1개 이상 수집
- profile/rule 수정 제안 수집
```

## 성공 기준

1차 도입 단계의 Pilot 성공은 "수치 달성"이 아니라 **"전사 확대 가능 여부를 판단할 수 있는가"**로 본다.
회고/노하우 기록은 강제하지 않으며, 부담 없는 .md 공유가 자연스럽게 일어나는지를 더 중요하게 본다.

### 참고지표 (목표 아님, D12 강등)

아래는 rollout 판단을 돕는 참고지표다. 강제 목표나 통과/실패 기준이 아니다.

| 참고지표 | 참고값 | 측정 방법 |
|---|---|---|
| Pilot 설치 성공률 | 80% 내외 | 설치 시도 대비 성공 인원 |
| 설치 평균 소요 시간 | 추세만 | 킥오프~첫 apply 성공까지 |
| 첫 업무 적용까지 걸린 시간 | 추세만 | 설치 완료~첫 산출물 |
| 공유 폴더 .md 업로드 수 | 추세만 | 공유 폴더에 모인 스킬/프롬프트 수 |
| 공유 .md 재사용 사례 | 추세만 | 남이 올린 .md를 선택해 적용한 건수 |
| 출처 표기율 | 높을수록 좋음 | 외부 자료 사용 건 대비 출처 표기 |
| 발견된 blocker 수 | 추세만 | 종료 시 미해결 blocker |

```text
참고지표는 통과 판정에 쓰지 않고 rollout 판단의 보조 근거로만 사용한다.
PR 노하우/주간 회고/Prompt DB 등록은 강제하지 않으므로 제출률을 통과 기준으로 두지 않는다.
공유 문화가 자리 잡은 뒤, 분기 거버넌스 리뷰에서 일부 지표를 다시 목표로 격상할지 결정한다.
```

### 완료(통과) 기준

```text
- 설치가 대상 OS/target에서 동작하고, 실패 시 원인과 workaround가 문서화된다.
- blocker가 0개이거나 모든 blocker에 workaround가 문서화된다.
- 비개발자도 설치 후 첫 업무를 수행한다.
- 공유 폴더에 .md가 실제로 모이고, 다른 사람이 선택해 적용한 사례가 1건 이상 나온다.
- profile/rule 수정 사항이 수집·반영된다.
- 전사 rollout 가능/불가를 근거와 함께 결정할 수 있다.
```

## 회고 질문

주간 회고와 종료 회고에서 아래 질문에 답한다. 답변은 Drive 운영 메모와 Pilot 회고 문서에 기록한다.

> 2주 종료 후 바로 쓸 수 있는 설문/인터뷰 문항지(설치 난이도·도구별 사용성·공유 폴더·보안 이해도):
> [docs/pilot-retrospective-survey.md](../../../pilot-retrospective-survey.md)

### 참가자 주간 회고

```text
1. 이번 주 agent로 실제로 끝낸 업무는 무엇인가? (1건 이상 구체적으로)
2. 설치/설정에서 막힌 지점은 어디였고, 어떻게 해결했나?
3. profile/rule/skill 중 도움이 된 것과 방해가 된 것은?
4. 다시 쓸 만한 프롬프트를 Prompt DB에 등록했나? 안 했다면 이유는?
5. 민감정보 입력 금지/출처 표기 룰을 지키기 어려웠던 상황이 있었나?
6. 같은 작업을 동료에게 추천할 때 가장 먼저 알려줄 한 가지는?
```

### 공용 계정 담당자 회고

```text
1. 담당 계정에서 자주 나온 사용 패턴은 무엇인가?
2. 공유하면 좋을 노하우/프롬프트가 쌓였나?
3. 계정 공유로 인한 충돌/대기/혼선이 있었나?
4. 금지 데이터 입력 시도나 위험 신호를 본 적이 있나?
```

### 종료 회고 (rollout 판단용)

```text
1. 전사로 확대해도 되는가? 그렇게 판단하는 핵심 근거는?
2. 확대 전에 반드시 고쳐야 할 blocker는 무엇인가?
3. 직무별(개발/기획/비개발/운영)로 더 필요한 asset은?
4. 비개발자 설치 경험에서 가장 큰 장벽은 무엇이었나?
```

## 실패 대응

Pilot 중 문제는 숨기지 않고 기록한다. 모든 실패는 agent가 재현 정보와 함께 피드백 `.md`로 정리해 Google Drive `AI-Knowhow/feedback/` 업로드를 유도하고, 대응 후 회고에 남긴다.

### 증상별 1차 대응

```text
- 설치 실패:
  SETUP_WIZARD.md "## 10. 설치가 안 될 때" 절차로 자가 복구 시도 →
  실패 시 agent가 명령과 dry-run 출력 전체를 피드백 `.md`로 정리하고 Drive 업로드를 안내.

- target 동작 불일치(같은 룰인데 도구별 결과가 다름):
  harness parity 이슈로 분류하고 어떤 target/profile에서 발생했는지 기록.

- 공용 계정 충돌/접근 불가:
  계정 담당자에게 먼저 알리고, 사용 시간/목적을 매핑표 기준으로 조정.

- 민감정보 입력 위험 신호:
  즉시 중단하고 governance 담당에게 보고. 입력된 적이 있으면 범위를 기록.
```

### 에스컬레이션 경로

```text
1차: 본인 자가 복구(가이드/troubleshooting)
2차: Google Drive feedback 업로드 + 계정 담당자/동료 지원
3차: agent-deploy maintainer(설치·adapter 문제) / governance 담당(룰·계정·보안 문제)
```

### blocker 처리 원칙

```text
- 모든 blocker는 "재현 절차 + 영향 받은 역할/OS/target + 임시 우회(workaround)" 형태로 기록한다.
- workaround가 있으면 통과 가능, 없으면 종료 산출물의 "설치 실패 목록"에 남긴다.
- 같은 blocker가 2명 이상에게 발생하면 우선순위를 높인다.
```

### Pilot 중단(abort) 기준

아래 중 하나라도 발생하면 Pilot을 일시 중단하고 대응 우선으로 전환한다.

```text
- 민감정보가 실제로 외부 도구에 입력된 사고가 발생한 경우
- 설치가 과반 참가자에게서 자가 복구 불가로 실패하는 경우
- 공용 계정 운영이 보안/정책상 지속 불가로 판단되는 경우
```

## Pilot 종료 산출물

