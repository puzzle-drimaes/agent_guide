# 10. Pilot 운영

## 목적

이 문서는 초기 Pilot 계획의 배경과 판단 기준을 남기는 계획 문서다.
실제 1차 실행은 별도 참가자 5명을 배정하는 폐쇄형 Pilot이 아니라, 준비된 bundle을 전사에 공개하고 실제 사용 피드백을 받는 **전사 베타**로 진행한다.

실행용 runbook은 아래 문서를 기준으로 한다.

```text
docs/pilot-runbook.md
```

전사 베타의 목표는 "완벽한 배포"가 아니라 "전사 rollout 가능/불가를 근거 있게 판단할 실제 사용 피드백을 빠르게 모으는 것"이다.

## 1차 실행 방향 변경

```text
- 별도 Pilot 참가자 5명 모집/배정은 하지 않는다.
- 참가자별 OS / target / profile도 사전 지정하지 않는다.
- D0 / D+7 / D+14 일정표와 스케줄러 기능은 1차 버전에서 제외한다.
- 운영자는 Windows/Linux에서 bundle 설치 알파 리허설을 먼저 수행한다.
- 전사 공지 후 사용자는 본인이 쓰는 AI 도구와 프로젝트에서 SETUP_WIZARD를 따라 설치한다.
- 사용 후 피드백은 agent가 `.md`로 정리해 Google Drive `AI-Knowhow/feedback/` 업로드를 유도한다.
```

## 운영자 알파 리허설 범위

전사 공지 전 운영자가 최소 Windows와 Linux에서 아래 lifecycle을 확인한다.

```text
- dry-run
- apply
- doctor
- update --dry-run
- repair --dry-run
- uninstall --dry-run
```

구체적인 명령 예시는 `docs/pilot-runbook.md`를 따른다.

## 전사 베타 피드백 방식

사용자는 별도 설문부터 작성하지 않는다. 설치 또는 실제 사용 후 본인이 쓰는 agent에게 피드백 정리 프롬프트를 실행하게 한다.

```text
- 배포 후 피드백은 agent가 `.md`로 정리해 Google Drive `AI-Knowhow/feedback/` 업로드를 먼저 유도한다.
- Drive 커넥터가 없으면 agent가 피드백 `.md` 내용을 만들어 주고, 사용자가 Drive 웹에서 업로드한다.
- blocker는 우선 workaround를 문서화하고, 재발 가능성이 높은 항목부터 bundle/profile에 반영한다.
- 피드백은 개인 평가가 아니라 운영 개선 자료로만 쓴다.
```

## 운영자가 취합할 항목

```text
- 설치 성공률
- 설치 평균 소요 시간
- project scope 설정 여부
- 도구별 의미 동등성
- 비개발자 profile/용어 이해도
- 공유 .md 업로드/재사용 사례
- 보안 위험 신호
- 반복 blocker와 workaround
```

## 성공 기준

1차 전사 베타의 성공은 "수치 달성"이 아니라 **"전사 확대 가능 여부를 판단할 수 있는가"**로 본다.
회고/노하우 기록은 강제하지 않으며, 부담 없는 .md 공유가 자연스럽게 일어나는지를 더 중요하게 본다.

### 참고지표 (목표 아님, D12 강등)

아래는 rollout 판단을 돕는 참고지표다. 강제 목표나 통과/실패 기준이 아니다.

| 참고지표 | 참고값 | 측정 방법 |
|---|---|---|
| 설치 성공률 | 추세만 | 설치 시도 대비 성공 건 |
| 설치 평균 소요 시간 | 추세만 | 킥오프~첫 apply 성공까지 |
| 첫 업무 적용까지 걸린 시간 | 추세만 | 설치 완료~첫 산출물 |
| 공유 폴더 .md 업로드 수 | 추세만 | 공유 폴더에 모인 스킬/프롬프트 수 |
| 공유 .md 재사용 사례 | 추세만 | 남이 올린 .md를 선택해 적용한 건수 |
| 출처 표기율 | 높을수록 좋음 | 외부 자료 사용 건 대비 출처 표기 |
| 발견된 blocker 수 | 추세만 | 종료 시 미해결 blocker |

```text
참고지표는 통과 판정에 쓰지 않고 rollout 판단의 보조 근거로만 사용한다.
PR 노하우/주간 회고/Prompt DB 등록은 강제하지 않으므로 제출률을 통과 기준으로 두지 않는다.
공유 문화가 자리 잡은 뒤, 분기 거버넌스 리뷰에서 일부 지표를 다시 목표로 격상할지 결정한다.
```

### 완료(통과) 기준

```text
- 설치가 대상 OS/target에서 동작하고, 실패 시 원인과 workaround가 문서화된다.
- blocker가 0개이거나 모든 blocker에 workaround가 문서화된다.
- 비개발자 사용자가 이해하기 어려운 표현과 절차가 수집된다.
- 공유 폴더에 .md가 실제로 모이고, 재사용 가능한 후보가 식별된다.
- profile/rule 수정 사항이 수집·반영된다.
- 전사 rollout 가능/불가를 근거와 함께 결정할 수 있다.
```

## 회고/설문 문서 위치

`docs/pilot-retrospective-survey.md`는 2주 고정 Pilot 설문 문항의 참고 자료로 남긴다. 1차 전사 베타에서는 `docs/pilot-runbook.md`의 피드백 정리 프롬프트를 우선 사용한다.

## 실패 대응

베타 중 문제는 숨기지 않고 기록한다. 모든 실패는 agent가 재현 정보와 함께 피드백 `.md`로 정리해 Google Drive `AI-Knowhow/feedback/` 업로드를 유도하고, 대응 후 회고에 남긴다.

### 증상별 1차 대응

```text
- 설치 실패:
  SETUP_WIZARD.md "## 10. 설치가 안 될 때" 절차로 자가 복구 시도 →
  실패 시 agent가 명령과 dry-run 출력 전체를 피드백 `.md`로 정리하고 Drive 업로드를 안내.

- target 동작 불일치(같은 룰인데 도구별 결과가 다름):
  harness parity 이슈로 분류하고 어떤 target/profile에서 발생했는지 기록.

- 공용 계정 충돌/접근 불가:
  계정 담당자에게 먼저 알리고, 사용 시간/목적을 매핑표 기준으로 조정.

- 민감정보 입력 위험 신호:
  즉시 중단하고 governance 담당에게 보고. 입력된 적이 있으면 범위를 기록.
```

### 에스컬레이션 경로

```text
1차: 본인 자가 복구(가이드/troubleshooting)
2차: Google Drive feedback 업로드 + 계정 담당자/동료 지원
3차: agent-deploy maintainer(설치·adapter 문제) / governance 담당(룰·계정·보안 문제)
```

### blocker 처리 원칙

```text
- 모든 blocker는 "재현 절차 + 영향 받은 역할/OS/target + 임시 우회(workaround)" 형태로 기록한다.
- workaround가 있으면 통과 가능, 없으면 종료 산출물의 "설치 실패 목록"에 남긴다.
- 같은 blocker가 2명 이상에게 발생하면 우선순위를 높인다.
```

### Pilot 중단(abort) 기준

아래 중 하나라도 발생하면 전사 베타를 일시 중단하고 대응 우선으로 전환한다.

```text
- 민감정보가 실제로 외부 도구에 입력된 사고가 발생한 경우
- 설치가 다수 사용자에게서 자가 복구 불가로 반복 실패하는 경우
- 공용 계정 운영이 보안/정책상 지속 불가로 판단되는 경우
```

## Pilot 종료 산출물

```text
- 설치 실패 목록
- profile 수정 목록
- rule 수정 목록
- 비개발자 UX 개선 목록
- 공용 계정 운영 이슈
- 참고지표 집계 결과(목표 대비 아님)
- feedback .md 요약
- 전사 rollout 가능/불가 판단과 근거
```
