# Git Commit Message Convention

## 메시지 구조

```text
[type] 제목 (기능 수정 내용을 요약)

1. 내용
- 버그 수정: 버그 현상 설명
- 기능 추가: 요구사항 설명

2. 수정 내역
- 수정한 기능에 대해 작성 (함수, 변수, 클래스 등)

3. 영향도
- 기존 코드 및 기능에 미치는 영향

jira : https://<YOUR_SPACE>.atlassian.net/browse/{TICKET-KEY}
```

## Commit line type

| 타입 | 설명 |
|---|---|
| `feat` | 새로운 기능 추가 / 기능 수정 |
| `fix` | 버그 수정 |
| `refactor` | 리팩토링, 기능 변화 없음 |
| `chore` | 설정, 빌드, 패키지 등 기능 외적인 작업 |
| `docs` | 문서 작성 또는 수정 |
| `test` | 테스트 코드 추가/수정 |

## 작성 규칙

1. 타입은 대괄호(`[]`)로 감싸고 뒤에 한 칸 띄운다.
2. 제목은 한글로 간결하게 작성한다.
3. 제목과 본문 사이에 빈 줄을 둔다.
4. 본문은 `1. 내용`, `2. 수정 내역`, `3. 영향도`로 구분한다.
5. 마지막에 Jira 티켓 링크를 적는다.
6. Jira 티켓이 없으면 agent가 임의로 만들지 말고 사용자에게 확인한다.

## 예외

팀에서 허용한 내부 문서 초안, spike, prototype은 아래 형식을 사용할 수 있다.

```text
jira : N/A (사유: 내부 문서 초안)
```

## AI agent 규칙

- 변경 내용이 여러 type에 걸치면 가장 중요한 목적 기준으로 type을 고른다.
- 기능 변경과 리팩토링이 섞이면 commit 분리를 제안한다.
- 문서 변경과 코드 변경이 독립적이면 commit 분리를 제안한다.
- 영향도는 실제 변경 범위를 기준으로 보수적으로 작성한다.
- commit message 작성 전 diff를 확인한다.
